1. Project Overview

SpecWeaver is a specification-first development lifecycle tool. AI agents draft specifications that are validated, reviewed, and then used to generate code and tests — all orchestrated by a pipeline engine with quality gates.

pip install -e ".[dev]"       # Install with dev dependencies
sw validate spec.md           # Validate a specification
sw run new_feature spec.md    # Run the full pipeline
python -m pytest -m unit -q   # Run unit tests

Package Map

PackagePurpose
cli/Typer CLI package — entry point + submodules (_core, _helpers, config, constitution, implement, lineage, pipelines, projects, review, standards, validation)
config/Project configuration, validation settings, database, domain profiles
context/Topology/dependency context for LLM prompts
drafting/Spec co-authoring with HITL interaction
flow/★ Pipeline engine (runner, gates, state, handlers)
graph/Project dependency graph (import analysis)
implementation/Code generation from specs
llm/LLM adapter layer (auto-discovery registry, Gemini, OpenAI, etc.), token/cost telemetry, prompt builder, mention scanner, artifact lineage
loom/★ Interaction layer — atoms, tools, commons
pipelines/YAML pipeline definitions
project/Project scaffolding, detection, Tach integration (SF-8), and constitution loader
review/LLM-powered spec and code review
validation/Rule-based validation (S01–S12, C01–C09, C12) + YAML sub-pipeline engine
workflows/L1-L5 domain-driven phases (planning, scenarios, drafting, pipelines)

CLI Pipeline Commands

Three commands expose the flow engine to users:

CommandDescription
sw run <pipeline> <spec_or_module>Execute a pipeline against a spec file or module name

Key flags

FlagEffect
--verbose / -vShow detailed handler output + full tracebacks on error
--jsonOutput NDJSON event stream (machine-readable, pipe to jq)
--project / -pSpecify project directory explicitly
--selectorTopology selector: direct, nhop, constraint, impact

Display Backends

The runner communicates progress via an on_event callback, with two display backends:

  • RichPipelineDisplay (default) — Live step list with ✅/❌/⠋/🅿️ checkmarks, elapsed time, gate annotations. Uses rich.live.Live.
  • JsonPipelineDisplay (--json) — Writes one JSON object per event to stdout (NDJSON). No Rich formatting.

Error Handling

  • Default: Friendly one-liner ([red]Error:[/red] message) + "Run with --verbose" hint
  • --verbose: Full traceback printed below the error message
  • KeyboardInterrupt: State saved to disk, resume instructions printed

Spec Argument Resolution

The spec_or_module argument is resolved differently per pipeline type:

  • validate_only: Treated as a file path (must exist)
  • new_feature: Treated as a module name, auto-derives specs/<name>_spec.md

6. Validation System

Validation Architecture

Two rule families. Each rule is a standalone class extending the Rule ABC with a self-declaring PARAM_MAP. Rules are executed via YAML-defined sub-pipelines (not hardcoded lists). Rules are kind-aware: the SpecKind enum (FEATURE, COMPONENT) selects presets that adjust headers, thresholds, and skip/enable behaviour.

7c. Domain Profiles

Domain profiles are named YAML pipeline configurations that tune validation thresholds for specific project types. Setting a profile selects a YAML pipeline which dictates the threshold rule cascade structurally.

Clear Separation:
  • YAML pipelines define structure — which rules run, in what order, with what base thresholds. Profiles select which YAML pipeline to use.
  • Local Pipeline Overrides located in .specweaver/pipelines/*.yaml provide runtime tuning — extending or overriding the base Pipeline.
  • CLI --set flags are ephemeral, session-only overrides with the highest precedence.

11. Key Design Decisions

DecisionRationale
Spec-first pipelineSpecs validated before code → catches issues at design time, not runtime
YAML pipeline definitionsNon-code configuration → easy to add new workflows without touching Python
Atom / Tool separationEngine and agents have different security boundaries → principle of least privilege
Commons shared layerPrevents cross-layer imports → atoms and tools stay independently testable
Language-agnostic interfacesQARunnerInterface ABC → future Go, Rust, TypeScript support
Role-based tool gatingImplementer gets fix=True, reviewer gets read-only → agents can't exceed authority
Gate-based flow controlRetry, loop-back, HITL parking → pipelines are self-healing
Feedback injectionFailed step findings enriched into next attempt's context → agents learn from mistakes
JSON state persistenceRuns survive process restarts → sw run --resume continues from last step
Standalone rule classesS01–S11 and C01–C08 are each one class → easy to add, test, and disable individually
Configurable thresholdsPer-project overrides via specweaver.yaml or --set CLI flag → no code changes needed
Kind-aware validationSpecKind enum (FEATURE/COMPONENT/CODE) selects rule presets → same rules adapt behaviour per spec type
Feature decompositionHITL-gated pipeline: draft feature → validate at feature thresholds → decompose into components
Constitution injectionProject-wide CONSTITUTION.md injected into every LLM call → consistent coding standards enforced by AI across review + implementation
Constitution walk-upWalk-up resolution from spec dir to project root → monorepo support with per-module overrides
Domain profiles as pure YAMLProfiles assign a YAML pipeline cascade which completely eliminates runtime configuration conflicts
Hardcoded profile dictsProfiles are Python dicts in profiles.py → versioned with code, quick-win over YAML/DB-stored profiles (deferred to 3.3b)
set-profile overwritesApplying a profile DELETEs all existing overrides first → predictable state, no leftover rules from previous profile
Self-describing LLM AdaptersAuto-discovery registry scans package at import → adding a new provider requires exactly one file without touching central maps
Cost Telemetry LayeringLookup: db overrides → adapter defaults. Ensures system defaults never overwrite user's DB overrides during upgrades
