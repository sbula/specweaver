# Feature 3.32a: Adaptive Assurance & Context Intelligence

## 1. Overview
Feature 3.32a overhauls SpecWeaver's contextual intelligence engine. By bundling three critically synergistic capabilitiesŌĆö**Adaptive Standards**, **Smart Scan Exclusions**, and **External Context Providers**ŌĆöwe transform the `StandardsAnalyzer` and `PromptBuilder` from a passive legacy mimic into a highly adaptive, configurable best-practices coach.

### Problems Solved
- **The Empty Repository Vacuum:** Greenfield projects (written in Java, Kotlin, Rust) currently provide zero legacy style data for the Agent to mimic, leading to standardless generation.
- **The "Garbage In, Garbage Out" Problem:** Scanning brownfield projects often results in mining autogenerated protobuf/gRPC codes or heavily minified bundles (like `node_modules` or `target/`), corrupting the Agent's implicit style guide.
- **The Opaque Infrastructure Problem:** Agents lack real-time visibility into opaque, external environmental systems (like 900-table database schemas or sprawling external REST APIs) without heavy API bindings.

## 2. Capabilities Introduced

### A. Adaptive Assurance Standards
Toggles the `StandardsAnalyzer` behavior based on project maturity. Configurable via `specweaver.toml`:
- **Mimicry (Brownfield Default):** Mines the existing codebase to ensure the Agent writes code that seamlessly matches the legacy developers' style.
- **Best Practice (Greenfield Default):** Injects a hardcoded, expertly curated set of idiomatic idioms (e.g., proper Kotlin immutable structures, Rust idiom linters) perfectly matching modern industry standards for languages that lack historical context in the repo.

### B. Smart Scan Exclusions (Tiered)
Protects the codebase scanner using a robust 3-tier exclusion boundary:
1. **Implicit Binary Extensions:** (`.pdf`, `.png`, `.jar`, `.so`) 
2. **Implicit Framework Patterns:** (`.git/`, `__pycache__/`, `node_modules/`, `target/`, `build/`)
3. **Explicit Repository Configuration:** Follows a dedicated `.specweaverignore` file.

### C. External Context Providers
Introduces a `.specweaver/context/` directory in the target repository. Users can drop arbitrary scripts (e.g., `dump_db_schema.py` or `fetch_live_api_types.sh`) into this protected directory. The `PromptBuilder` will natively execute these scripts, capturing `stdout`, and piping those living architectural truths directly into the LLM context.

## 3. Architecture & Integration Plan

### Touched Components
- `src/specweaver/core/config/settings.py`: Expanding configuration schemas for `specweaver.toml` (exclusion rules, standard enforcement targets).
- `src/specweaver/assurance/standards/analyzer.py`: Incorporating the Smart Scan Exclusion lists natively to bypass unreadable directories and parsing injection modes.
- `src/specweaver/infrastructure/llm/prompt_builder.py`: Refactoring the context payload payload injector to query `specweaver/context/` providers dynamically before LLM submission.

### Boundary Constraints
- **Security Limitation:** External Context Providers (`context/providers.py`) MUST be strictly bounded to local `.specweaver/` locations protected by `FolderGrant` to prevent LLM Remote Code Execution hacks. They are read-only injections.
- **Tach Validation:** We cannot introduce loops between the `core/config` and the `infrastructure/llm`; the `PromptBuilder` accepts injected configurations unidirectionally.
